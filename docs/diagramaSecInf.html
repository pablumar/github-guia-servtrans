Busqueda de Pacientes

Esta página describe los fragmentos de secuencia usados para la búsqueda y obtención de informes clínicos (por ejemplo, resultados de laboratorio, informes de imagen o informes de alta hospitalaria) y se señalan a continuación las posibles opciones de implementación.

Visión de Negocio

El diagrama muestra dos variantes principales para recuperar informes: Search and Get y Fetch. En la primera, el solicitante realiza una búsqueda de informes (obteniendo un resumen o datos mínimos) y, posteriormente, selecciona cuáles descargar completamente. En la segunda, la recuperación del informe se realiza directamente sin pasar por la fase de presentación de datos mínimos.

A continuación se ilustra el fragmento “Search and Get” o “Fetch” Report(s):

Visión de Aplicación

Según el diagrama, el flujo Search and Get incluye los siguientes pasos:

  1. Search Reports: el Requestor solicita una búsqueda de informes en el Requesting NCP, que a su vez delega en el Responding NCP. Se devuelve información mínima (por ejemplo, metadatos de cada informe).
  2. Select Report(s): el Requestor escoge uno o varios informes de la lista devuelta.
  3. Get Report(s): se solicitan los informes seleccionados al Responding NCP a través del Requesting NCP. El Responding NCP devuelve el contenido completo de los informes.

En el flujo Fetch, el Requestor conoce de antemano el/los informe(s) a solicitar y no precisa un paso intermedio de búsqueda de metadatos. Simplemente envía una petición “Fetch Reports” y recibe de vuelta el contenido completo.

Petición: Search / Fetch Reports

Para implementar estos flujos, se recomienda el uso de DocumentReference en FHIR. Si se opta por Search and Get, el proveedor debería soportar parámetros que permitan:

Parámetro Consumer Provider searchParameter type Notas
Tipo de documento (Document type) SHALL SHALL type token Filtra por tipo de informe (p.ej. LOINC para resultados de laboratorio).
Tipo de contenido (Type of content) MAY SHALL contenttype token Permite filtrar por formato (PDF, FHIR, CDA, etc.).
Categoría / Estudio (Study type) MAY SHALL category token Ej.: laboratorio, microbiología, radiología.
Identificador del paciente SHALL SHALL patient reference Normalmente patient.identifier para enlazar con el sistema de origen.
Fecha SHALL SHALL date date Filtra por la fecha de creación del DocumentReference.
Fecha de creación MAY MAY creation date Permite buscar por DocumentReference.content.attachment.creation.

Si en cambio se emplea Fetch, puede omitirse la fase de búsqueda y solicitar directamente el documento o documentos requeridos cuando se conoce su identificador.

Respuesta: Search / Fetch Reports

Si se usan parámetros de búsqueda, el servidor devolverá un Bundle de tipo searchset con los DocumentReference encontrados. Cada uno puede incluir o referir varias representaciones (PDF, HL7 CDA, HL7 FHIR, etc.) del mismo documento.

  • Si el contenido se aloja en content.data, estamos ante el caso Fetch (documento embebido).
  • Si la referencia está en content.url, se usa el patrón Search and Get, solicitando el contenido completo a la URL indicada (por ejemplo, un GET /Binary/id o GET /Bundle/id).

Petición: Obtención de Informes por ID

En Search and Get, una vez seleccionado el informe, se puede recuperar el documento usando la URL devuelta. Dependiendo del tipo de contenido:

  1. Si es un documento FHIR (por ejemplo Bundle), se hace GET /Bundle/[id].
  2. Si es un formato no FHIR (PDF, CDA, etc.), se hace GET /Binary/[id].

Respuesta: Obtención de Informes por ID

Para documentos FHIR, se devuelve un Bundle que cumple con el perfil o especificación que corresponda al tipo de informe. Para documentos no FHIR, se devuelve un recurso Binary con los datos binarios (PDF, CDA, etc.).

Códigos de Respuesta HTTP ante Errores

HTTP Response Descripción
401 Unauthorized Se requiere autorización para la interacción solicitada
404 Not Found Tipo de recurso no soportado o no es un end-point FHIR
410 Gone El recurso fue eliminado o ya no está activo

Con estos pasos y opciones se cubren ambos flujos (búsqueda y obtención, o recuperación directa), siguiendo el esquema descrito en el diagrama anterior.
